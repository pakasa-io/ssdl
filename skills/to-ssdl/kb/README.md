# Knowledge base — extracted source facts, per `.ssdl` element

A **fact-extraction layer**. Extraction is **lazy and scoped** (see below): Phase 2 records the app-wide shared
contracts, and Phase 5 extracts each screen's facts just-in-time — into one YAML card per `.ssdl` element. Each
section is then authored from its card's `facts` **plus** the SSDL spec slice — so the output reflects **real
contracts** (real attribute names, types, statuses, constraints), not invented ones.

```
source material  ──extract──▶  KB facts   (WHAT is real)
SSDL spec slice  ───────────▶  HOW to express it
                   agent fuses ▶  grounded .ssdl
```

## Layout

Mirrors the topological (define-before-reference) order — tier folders `0-frame` … `7-verification`, one card per
element. (Sections today; components and directives extend the same way.)

## Card schema

```yaml
element: MODEL
kind: section
order: 13
tier: 2-data-and-contracts
extracted_from: []      # provenance — which source artifact(s) the facts came from
facts:                  # the extracted facts, shaped per element — a full descriptor per item
  request: {}           # <Model>: { applies_to: [screens], attrs: [ { attr, type, format, required, enum, min/max, pattern, example, sensitive } ] }
  response: {}          # <Model>: { applies_to: [screens], attrs: [ { attr, type, format, nullable, enum, example, sensitive, relation } ] }
  entities: {}          # <Name>:  { applies_to: [screens], key, attrs: [...], relations: [ { to, kind } ] }
feeds: [DATA, UI, VALIDATION, BUSINESS_RULES, ACTIONS, FLOW, ANALYTICS, ACCEPTANCE]   # who consumes these facts
```

`facts` is shaped per element (MODEL→`request`/`response`/`entities`, API→`operations`, VALIDATION→`constraints`,
ERRORS→`responses`, ROUTE→`path`/`params`, …). Each item is a **full descriptor** — a MODEL attribute carries its
constraints, enum, example, and sensitivity, so VALIDATION / UI / ANALYTICS each lift exactly what they need
without re-reading the source. `feeds` is the **reverse of the dependency edges** — the sections that consume
this card's facts.

## Extraction is lazy and scoped (never a one-shot pass)

Extract facts **only for what is in play**, mirroring the skill's lazy spec-slice loading — never sweep the whole
source into the KB up front:

- **Phase 2 — shared contracts only.** Catalog the source artifacts, then extract the **app-wide** facts the
  `shared/` layer needs: entities, base API conventions, the global error catalog (fill `facts` + `extracted_from`;
  update `_index.yaml`). OpenAPI is the prime source here.
- **Phase 5 — screen-scoped, just-in-time.** As each screen is built, extract **only that screen's** facts into its
  cards — its `MODEL`/`API`/`VALIDATION`/`ERRORS` from the contract, its `STATES`/`FLOW`/`A11Y`/`ACCEPTANCE` from
  the PRD/design — reusing shared contracts already captured.

**Rule (strict):** pre-extracting facts for elements or screens not yet in scope is a defect. Anything missing or
ambiguous in the source goes to `30-OPEN_QUESTIONS.gaps`.

## Consumption (Phase 5)

Author each section from its card's `facts` + the section's SSDL spec slice. Dependency order + `feeds` make a fact
flow forward — one extracted attribute propagates to every section that consumes it:

```
email { format: email, required: true }            (extracted into MODEL.facts.request)
  ├─▶ MODEL       $email!: Email := ""
  ├─▶ VALIDATION  VAL-01: !matchesEmail($email) => "..."
  ├─▶ UI          #email: Input bind:$email keyboard:email
  └─▶ ANALYTICS   privacy.never_send: [$email]
```

## `_index.yaml`

A fast reverse lookup over the extracted facts — `sources` / `operations` / `entities` / `attributes` → the cards
that hold or use them. Query it before authoring ("where are `User`'s facts, and which sections use them?").

## Scope & growth

Facts are global (operations, entities) or screen-scoped — **each fact carries its own `applies_to`**, since one
card holds facts spanning many screens. `applies_to` is populated **retrospectively** — a fact is often extracted
before its consumers are known (a shared `User` entity, an API operation), so each card's `applies_to` is appended
as the screens that use the fact are built. The KB **accretes** as more source
is processed and more screens are built — facts captured once are reused, so the generated corpus stays consistent
with the real backend.

## Maintenance

The scaffold, the per-element `facts` shapes, and the `feeds` edges are generated by [`_build.py`](_build.py) — the
single source of truth for the card schema. Change `_build.py` and re-run `python3 skills/to-ssdl/kb/_build.py`;
do not hand-edit the empty scaffold. The skill fills `facts` / `extracted_from` / `context` at extraction time.
